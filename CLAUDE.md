Default to using Bun instead of Node.js.

- Use `bun <file>` instead of `node <file>` or `ts-node <file>`
- Use `bun test` instead of `jest` or `vitest`
- Use `bun build <file.html|file.ts|file.css>` instead of `webpack` or `esbuild`
- Use `bun install` instead of `npm install` or `yarn install` or `pnpm install`
- Use `bun run <script>` instead of `npm run <script>` or `yarn run <script>` or `pnpm run <script>`
- Use `bunx <package> <command>` instead of `npx <package> <command>`
- Bun automatically loads .env, so don't use dotenv.

## APIs

- `Bun.serve()` supports WebSockets, HTTPS, and routes. Don't use `express`.
- `bun:sqlite` for SQLite. Don't use `better-sqlite3`.
- `Bun.redis` for Redis. Don't use `ioredis`.
- `Bun.sql` for Postgres. Don't use `pg` or `postgres.js`.
- `WebSocket` is built-in. Don't use `ws`.
- Prefer `Bun.file` over `node:fs`'s readFile/writeFile
- Bun.$`ls` instead of execa.

## Testing

Use `bun test` to run tests.

```ts#index.test.ts
import { test, expect } from "bun:test";

test("hello world", () => {
  expect(1).toBe(1);
});
```

## Project: Web3 Indexer

This project is a web3 indexer built with **Pipes SDK** from Subsquid.

Documentation: https://beta.docs.sqd.dev/en/sdk/pipes-sdk/evm.autogenerated/quickstart

### Installation

```bash
bun add @subsquid/pipes @clickhouse/client
```

### Pipes SDK Architecture

The SDK uses a pipeline model with four components:

1. **Portal Source** — Streams blockchain data from SQD Portal
2. **Event Decoder** — Filters and decodes events using ABIs
3. **Transformer** — Maps decoded events to your data structure
4. **Target** — Persists data to database with rollback support

### Example: ERC20 Transfers

```typescript
import { evmPortalSource, evmDecoder, commonAbis } from '@subsquid/pipes';
import { clickhouseTarget } from '@subsquid/pipes';

await evmPortalSource({
  portal: 'https://portal.sqd.dev/datasets/ethereum-mainnet',
})
  .pipe(evmDecoder({
    range: { from: 'latest' },
    events: { transfers: commonAbis.erc20.events.Transfer },
  }))
  .pipe({
    transform: (data) =>
      data.transfers.map((t) => ({
        block_number: t.block.number,
        token: t.contract,
        from: t.event.from,
        to: t.event.to,
        amount: t.event.value.toString(),
      })),
  })
  .pipeTo(clickhouseTarget({...}))
```

### Key Concepts

- **Portal URL** — Points to data source (e.g., `ethereum-mainnet`)
- **Range** — Controls where indexing begins (supports `"latest"` for real-time)
- **Events** — Specify which events to decode with their ABI definitions
- **Rollback handling** — Implement `onRollback` for chain reorg consistency

### ClickHouse Schema Pitfall

Never use `UInt16` for `chain_id` columns. EVM chain IDs can exceed 65,535 (e.g., Katana is 747474). ClickHouse silently overflows without error, causing data to be stored under the wrong ID (`747474 % 65536 = 26578`). Always use `UInt32` or larger for chain IDs.
